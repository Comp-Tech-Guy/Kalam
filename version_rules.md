# Versioning

## Scheme

This project follows [Semantic Versioning](https://semver.org/): **MAJOR.MINOR.PATCH**

- **MAJOR** — Breaking changes (UI overhaul, tool schema incompatibility, dropped features)
- **MINOR** — New features, new tool integrations, added profile capabilities
- **PATCH** — Bug fixes, performance improvements, dependency updates

Pre-release suffixes (e.g. `0.2.0-alpha.1`, `0.2.0-rc.1`) are used for testing builds.

## Where versions live

| File | Purpose | Updated by |
|------|---------|-----------|
| `app/package.json` | npm registry, Tauri frontend | Manually before release |
| `app/src-tauri/tauri.conf.json` | Tauri bundle metadata | Manually + overwritten by CI |
| `latest.json` (GitHub Release asset) | Auto-update manifest | Auto-generated by CI |

**Before every release:** bump both `package.json` and `tauri.conf.json` to the same version.

## Release flow

1. Bump versions → `git commit -m "vX.Y.Z"`
2. Tag: `git tag vX.Y.Z && git push origin vX.Y.Z`
3. CI triggers, builds, creates draft release
4. Manually publish the draft on GitHub Releases page
5. App detects update on next start

## Zero-based versioning

Until the first stable release (`1.0.0`), the major version stays at `0`:
- **`0.1.0`** — initial development (current)
- **`0.2.0`**, `0.3.0`, ... — incremental features and fixes
- **`1.0.0`** — first stable, once the app is feature-complete for general use
